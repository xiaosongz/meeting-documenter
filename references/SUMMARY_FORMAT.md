# Meeting Summary Format Specification

This document defines the output format for meeting summaries generated by the meeting-documenter skill.

> **CRITICAL OBSIDIAN RENDERING RULE**: Always include a **blank line before tables**. Obsidian requires this for proper table rendering in both Live Preview and Reading modes. Tables directly after bold text, headings, or other inline elements will NOT render as tables.

## File Naming Convention

**Pattern**: `YYYY-MM-DD-HHMM MeetingName.md` (SPACE before title)

**Examples**:
- `2026-01-19-1500 ProjectAlpha-Standup.md`
- `2026-01-15-1302 Team-Weekly-Sync.md`
- `2026-01-07-1200 ProjectBravo-Q1-Planning.md`

**Rules**:
- Date with hyphens: `YYYY-MM-DD`
- Time 4-digits: `HHMM` (rounded to nearest minute for meetings)
- **SPACE separator** before title (NOT hyphen)
- Use hyphens to separate words within the meeting name
- Keep name concise but descriptive
- Include key identifiers (project name, team name, main topic)

**CRITICAL — date format examples:**
- ✅ `2026-01-19-1200 weekly-checkin.md`
- ❌ `20260119-weekly-checkin.md`
- ❌ `2026-01-19-1200-weekly-checkin.md`

## Link Styles

All `[[Name]]` examples in this document are written in the default Obsidian wikilink form. The actual output form is controlled by the `LINK_STYLE` env var (see `SKILL.md § Configuration`). Substitute every link example below using this table:

| `LINK_STYLE` | Attendee in YAML | Inline person ref | Transcript/recording source |
|---|---|---|---|
| `wikilink` (default) | `"[[Alex Kim]]"` | `[[Alex Kim]]` | see code fence below |
| `markdown` | `"[Alex Kim](People/Alex%20Kim.md)"` | `[Alex Kim](People/Alex%20Kim.md)` | `[Full Transcript](raw/2026-05-27-1030%20Sync-Transcript.md)` |
| `plain` | `Alex Kim` (string, no link) | `Alex Kim` | `raw/2026-05-27-1030 Sync-Transcript.md` |

**Wikilink transcript/recording source** uses Obsidian's display-text aliasing — the `|` is the alias separator and **must be emitted bare**, not escaped. (The table cell above can't carry a literal `|` without breaking the column delimiter, so the form is shown here instead):

```text
[[raw/2026-05-27-1030 Sync-Transcript|Full Transcript]]
```

For `markdown` style, the agent must know the path to each linked note. For attendees, default to `People/<Name>.md` if no other layout is configured. For `plain` style, no clickable navigation is produced — pick this only if the user's note system doesn't support either link form.

`LINK_STYLE` also affects `references/QUALITY_CHECKLIST.md` link-form verification rules.

## YAML Frontmatter

### Required Fields

```yaml
---
title: "Meeting Title"
date: 2026-01-19
type: meeting
tags:
  - "#meeting"
  - "#project-tag"
attendees:
  - "[[Name 1]]"
  - "[[Name 2]]"
meeting_type: Project Check-in | Team Meeting | Planning Session | etc.
meeting_outcome: decision | update | planning | blocked | cancelled
---
```

When `LINK_STYLE=wikilink`, use wikilinks for attendees to enable Obsidian graph navigation and create stub People Notes if they don't exist. For `markdown` or `plain` styles, see the Link Styles section above.

### Optional Fields

```yaml
project: ProjectFolderName  # Links to your projects directory
location: Zoom (meeting ID) | Room Name | etc.
duration: 30 min | 1 hour | etc.
```

### Tag Guidelines

| Always Include | Context-Specific |
|----------------|------------------|
| `#meeting` | Project tag (e.g., `#project-alpha`) |
| | Team tag (e.g., `#engineering`, `#ops`) |
| | Topic tag (e.g., `#planning`, `#review`) |

## Document Structure

### Header Section

```markdown
# Meeting Title (Descriptive)

**Date:** Month DD, YYYY
**Time:** H:MM PM - H:MM PM (timezone if relevant)
**Meeting Type:** Type of meeting
```

### Executive Summary

2-3 sentences capturing:
- Main purpose/context of meeting
- Key outcome or decision
- Next steps at high level

```markdown
## Executive Summary

[2-3 sentences. Focus on outcomes, not process. Answer: "What happened and why does it matter?"]
```

### Key Decisions

Bulleted list of decisions made during the meeting.

```markdown
## Key Decisions

- [Decision 1 - clear and actionable]
- [Decision 2 - include context if not obvious]
```

**Guidelines**:
- Only include actual decisions (not discussions)
- Make each decision self-contained
- Include owner/responsible party if relevant

### Action Items

Grouped by person with checkbox format.

```markdown
## Action Items

### [Person Name]
- [ ] Task description [due:: YYYY-MM-DD] [project:: [[ProjectName]]] [assignee:: Person Name]
- [ ] Another task [due:: YYYY-MM-DD] [assignee:: Person Name]

### [Another Person]
- [ ] Their task [due:: YYYY-MM-DD] [assignee:: Another Person]
```

**Guidelines**:
- Use checkbox format `- [ ]` for Obsidian task tracking
- Add `[due:: YYYY-MM-DD]` inline field when a deadline is mentioned or can be inferred
- Add `[project:: [[ProjectName]]]` inline field to link tasks to their project
- Add `[assignee:: Person Name]` on every action item — enables per-person task queries: `TASK WHERE assignee = "Name"`
- Make tasks specific and verifiable
- Assign to "Team" or "TBD" if owner unclear

### Parking Lot

Items explicitly tabled during the meeting — no owner assigned yet.

```markdown
## Parking Lot

- [Item deferred for future discussion]
```

**Guidelines:**
- Only include items explicitly tabled ("let's park that", "come back to", "table for now", "discuss next time")
- These items have NO owner — if an owner is identifiable, it's an Action Item instead
- Distinguish from Follow-Up Items: parking lot = explicitly deferred during meeting; follow-up = general future agenda
- Include "No items tabled." if nothing was explicitly deferred

### Topics Discussed

Summary paragraphs with timestamps (if available from transcript).

```markdown
## Topics Discussed

### 1. Topic Name [MM:SS]
[1-2 paragraph summary of discussion, key points raised, any sub-decisions]

### 2. Another Topic [MM:SS]
[Summary paragraph]
```

**Guidelines**:
- Number topics sequentially
- Include timestamp from transcript if available
- Summarize, don't transcribe verbatim
- Note disagreements or alternative views if significant

### Follow-Up Items

Items requiring future attention but not immediate action.

```markdown
## Follow-Up Items

- [Item for next meeting]
- [Topic to revisit later]
- [Information needed from external source]
```

### Footer

Render the `Source` line and `See Also` links per `LINK_STYLE` — substitute every `[[...]]` example below according to the Link Styles table at the top of this file.

```markdown
---

**Document Status:** Meeting Summary
**Compiled By:** [Author] (via Claude Code)
**Source**: [[raw/YYYY-MM-DD-HHMM Name-Transcript|Full Transcript]] | [[recordings/YYYY-MM-DD-HHMM Name.ogg|Recording (duration, OGG)]]

## See Also

- [[Related Note 1]]
- [[Related Note 2]]
```

## Quality Criteria

### Content Quality
- [ ] Executive summary is concise (2-3 sentences max)
- [ ] All decisions are actual decisions, not discussions
- [ ] Action items have owners assigned
- [ ] Action items are specific and verifiable
- [ ] Topics are summarized, not transcribed verbatim
- [ ] Parking Lot section present (use "No items tabled." if nothing deferred)

### Format Quality
- [ ] YAML frontmatter is valid
- [ ] `meeting_outcome` field present and set to one of: `decision`, `update`, `planning`, `blocked`, `cancelled`
- [ ] Attendees use the link form set by `LINK_STYLE` (see Link Styles section above)
- [ ] All action items include `[assignee:: Person Name]`
- [ ] All required sections present
- [ ] Timestamps included where available
- [ ] Source link to transcript is correct
- [ ] All inline person refs and source links match `LINK_STYLE`
- [ ] **CRITICAL: Blank line before every table** (CommonMark + Obsidian both require this for proper table rendering)

### Integration Quality
- [ ] Tags are lowercase with hyphens
- [ ] Project field matches folder name exactly
- [ ] Attendee names are consistent with existing notes
- [ ] Related links are valid

## Special Cases

### Short Meetings (< 15 min)
- May combine Topics Discussed into a single section
- Follow-Up Items section optional if none identified

### Large Meetings (> 10 attendees)
- List attendees with roles/departments
- May add "~X total participants" if exact count uncertain
- Group action items by team/department if applicable

### Recurring Meetings
- Reference previous meeting notes in "See Also"
- Note any carryover items from previous meeting
- Track status of previously assigned action items
